## 1. 프로젝트 개요 (Overview)

### 1.1 프로젝트 목적
Deal Architect는 단순한 영업 관리 도구(CRM)가 아닙니다. 프리세일즈(PreSales) 엔지니어가 고객의 구매 여정에 맞춰 기술적, 비즈니스적 의사결정을 내릴 수 있도록 돕는 **전략 설계 및 의사결정 지원 시스템**입니다.

### 1.2 해결하려는 문제
- **단계와 맥락의 불일치**: 영업 단계는 진행되었으나 고객의 실제 구매 준비도는 파악되지 않는 문제.
- **파편화된 데이터**: 고객 인터뷰(Discovery), 기술 검증(PoC), 경쟁 분석 데이터가 서로 연결되지 않고 흩어져 있는 현상.
- **판단 근거의 부재**: "왜 이길 수 있는가?"에 대한 질문에 사실 기반(Fact-based)이 아닌 감으로 대답하는 관행.

### 1.3 대상 사용자
- 솔루션 아키텍트 및 프리세일즈 엔지니어
- 기술 영업 대표 (Account Executive)
- 딜의 기술적 승리 전략을 검토하는 프리세일즈 매니저/리더

### 1.4 한 문장 요약
**"고객의 구매 여정 신호를 해석하여 승리 가설을 설계하고 검증하는 프리세일즈 전용 전략 플랫폼"**

---

## 2. 방법론적 기반 (Methodology Foundation)

### 2.1 고객 여정 기반 프리세일즈 접근법
영업은 우리가 파는 과정이 아니라, **고객이 사는 과정을 돕는 것**입니다. 본 앱은 프리세일즈 활동을 고객의 구매 여정(Customer Buying Journey) 단계와 동기화시킵니다.

### 2.2 기존 영업 단계 모델의 한계
기존 모델은 내부적인 '태스크 완료'를 기준으로 단계를 나눕니다. 하지만 고객은 단계와 상관없이 문제를 겪거나 감정의 변화를 일으킵니다. Deal Architect는 이러한 '고객 신호'를 중심으로 작동합니다.

### 2.3 “단계”가 아닌 “맥락 정합성” 중심 사고
현재 딜이 '평가' 단계에 있다면, 단순히 PoC를 마친 것이 아니라 고객이 "왜 우리가 이길 수 있는가?"라는 질문에 대한 답을 찾는 맥락에 있음을 의미합니다.

### 2.4 프리세일즈의 역할 재정의
프리세일즈는 제품을 설명하는 사람이 아니라, **고객의 비즈니스 문제를 기술적 해결책으로 번역하고 승리 가설을 입증하는 전략가**입니다.

## 3. CSA Framework 개요

### 3.1 CSA Framework 정의
**CSA (Customer Signal & Activity)** 프레임워크는 고객으로부터 발생하는 신호를 수집하고, 이를 분석하여 최적의 프리세일즈 활동을 도출하는 본 앱의 핵심 엔진입니다.

### 3.2 CSA의 핵심 구성 요소
- **Signal**: 고객의 행동, 문제, 감정, 접점 데이터.
- **Analysis**: 수집된 신호를 바탕으로 도출된 JTBD(Jobs To Be Done)와 성공 기준(Success Criteria).
- **Activity**: 분석 결과를 바탕으로 실행하는 구체적인 프리세일즈 작업(Solution Map, Fit Analysis 등).

### 3.3 고객 여정 단계 정의
1. **인식 (Awareness)**: "이 딜을 더 볼 가치가 있는가?" (Go/No-Go 판단)
2. **고려 (Consideration)**: "리소스를 써도 되는가?" (적합성 검증)
3. **평가 (Evaluation)**: "왜 우리가 이길 수 있는가?" (경쟁 우위 확보)
4. **구매 (Purchase)**: "어떻게 이길 것인가?" (최종 의사결정 유도)

### 3.4 고객 여정과 프리세일즈 활동의 매핑
각 단계마다 활성화되는 도구(Discovery, Qualification, Solution Map, Competitive Fit, Win Strategy)가 다르며, 이는 단계별 핵심 질문에 답하기 위해 설계되었습니다.

## 4. 전체 앱 구조 개요 (High-Level Architecture)

### 4.1 사용자 흐름 (User Flow)
1. **Deal 생성**: 기본 정보 및 목표 수주일 설정.
2. **Discovery**: 고객 신호 입력 및 AI를 통한 JTBD/SC 도출 (상시 엔진).
3. **Stage-based Tools**: 현재 단계에 따라 Qualification, Map, Competitive 분석 수행.
4. **Strategy Report**: 분석 데이터를 종합하여 TWS(기술 승리 전략) 또는 DWS(딜 승리 전략) 생성.
5. **Deal Close**: 최종 결과 기록 및 Lessons Learned 수집.

### 4.2 주요 화면 구성
- **Deals List**: 파이프라인 전체 조망 및 상태 관리.
- **Deal Context**: 상단 Stage Anchor를 통한 현재 맥락 고정.
- **Feature Workspace**: 각 활동(Discovery, Map 등)이 이루어지는 독립적인 작업 영역.

### 4.3 모듈 간 관계
- **Store (Firebase)**: 모든 딜 데이터의 중앙 저장소.
- **API (Gemini via GAS)**: 비정형 데이터를 정형화하고 전략을 초안하는 지능형 엔진.
- **Views**: UI 컴포넌트 기반의 반응형 인터페이스.

---

## 5. Deal Lifecycle & 고객 여정 기반 Workflow

### 5.1 Stage 전환(Trigger) 설계 원칙
단계 전환은 자동이 아닌 **명시적 선언**입니다. 핵심 질문에 답할 수 있는 근거(Evidence)가 준비되었을 때만 사용자가 버튼을 눌러 이동할 수 있습니다.

---

## 6. 핵심 프리세일즈 활동 프레임워크

### 6.1 Discovery (고객 맥락 수집)
단순한 초기 정보 수집이 아니라 전 단계에 걸쳐 고객의 신호를 업데이트하는 활동입니다.

### 6.2 Deal Qualification (기회 적합성 판단)
비즈니스(BANT)와 기술(Tech Readiness) 측면에서 딜의 건전성을 평가합니다.

### 6.3 Competitive Fit vs Win Strategy (중요)
- **Competitive Fit**: 사실 기반의 비교 분석 (분석 데이터 생성).
- **Win Strategy**: 분석 데이터를 바탕으로 한 설득 논리와 대응 시나리오 설계 (전략 수립).

---

## 7. 주요 기능 상세 설명

- **Solution Map**: 고객의 환경을 시각화하고 우리 솔루션의 점유율과 Pain Points를 트리맵으로 표현.
- **Fit Score**: 0~100점 사이의 점수를 통해 딜의 성숙도를 시각화하며, 이는 "결론"이 아닌 "팀 간의 대화 도구"로 활용됩니다.

---

## 8. AI 활용 설계

### 8.1 AI의 역할 범위
- **번역가**: 고객의 파편화된 진술을 JTBD로 번역.
- **분석가**: 방대한 경쟁 데이터를 표 형식으로 요약.
- **초안 작성자**: 전략 보고서의 구조를 잡고 논리적 가설 제시.

### 8.2 사람이 판단해야 하는 영역
- **최종 점수 확정**: AI의 추천 점수를 검토하고 실무적인 판단 가미.
- **전략의 실행**: 도출된 메시지를 고객에게 전달하고 반응을 살피는 활동.

## 10. 이 README의 사용 목적 (For AI Readers)

### 10.1 AI가 이해해야 할 핵심 전제
이 앱은 **"판단의 흐름"**이 **"작업의 흐름"**보다 우선합니다. 모든 UI와 로직은 사용자가 의사결정을 내리는 데 필요한 데이터를 최적의 타이밍에 제공하도록 설계되었습니다.

### 10.2 재구현 시 유지 원칙
1. **Stage Anchor의 위계**: 현재 단계와 핵심 질문은 항상 최상단에 노출되어야 함.
2. **Discovery의 상시성**: Discovery는 초기 메뉴가 아니라 모든 단계에서 수정 가능해야 함.
3. **사실과 전략의 분리**: 경쟁 분석(Fact)과 승리 전략(Strategy) 모듈의 역할 책임을 혼용하지 말 것.
4. **명시적 전환**: 단계 이동은 시스템이 아닌 사용자의 판단에 의한 명시적 트리거로 발생해야 함.

## 11. 고객 여정 데이터 모델

### 11.1 고객 여정 단계 정의
고객 구매 여정은 인식, 고려, 평가, 구매의 4단계로 구성됩니다. 각 단계는 고객이 해결하고자 하는 고유한 비즈니스 맥락과 의사결정 기준을 가집니다.

### 11.2 단계별 고객 데이터 항목
효과적인 프리세일즈 전략 수립을 위해 다음 5가지 핵심 데이터를 상시 수집합니다.
- **고객 행동**: 고객이 현재 문제를 해결하기 위해 취하고 있는 구체적인 액션.
- **고객 문제 (Pain)**: 현재 방식의 한계로 인해 발생하는 손실, 위험 및 장애 요인.
- **고객 감정**: 변화에 대한 고객의 심리적 동기, 불안 요소 및 기대치.
- **고객 접점 / 채널**: 정보가 흐르고 의사결정이 일어나는 주요 경로.
- **JTBD (Jobs To Be Done)**: 고객이 궁극적으로 달성하고자 하는 목적지(Outcome).

### 11.3 프리세일즈 To-Do 구조
모든 활동은 현재 단계의 '핵심 질문'에 답하기 위한 목적을 가집니다. Task 중심이 아닌 Evidence(증거) 중심으로 리스트가 관리됩니다.

### 11.4 단계별 정보 정확도 차이
초기 단계(인식)에서는 정성적이고 불확실한 신호가 많으나, 단계가 진행됨에 따라 정량적이고 구체적인 데이터(KFR, 예산 등)로 구체화되며 정보의 충실도가 높아집니다.

## 12. 평가 & 판단 로직

### 12.1 Qualification의 목적
단순한 점수 매기기가 아니라, "우리가 이 딜에 리소스를 투입할 가치가 있는가?"와 "승리할 수 있는 환경인가?"를 조기에 판단하여 세일즈 리소스를 최적화하는 데 있습니다.

### 12.2 Biz Fit / Tech Fit 개념
- **Biz Fit (BANT)**: 예산, 권한, 필요성, 일정을 기준으로 비즈니스적 성숙도를 평가합니다.
- **Tech Fit**: 기술 요구사항, 아키텍처 호환성, 운영 가능성을 기준으로 솔루션의 기술적 정합성을 평가합니다.

### 12.3 Competitive Fit의 위치
경쟁 분석은 '전략'을 세우기 전의 '사실(Fact)' 수집 단계입니다. 객관적인 비교표를 통해 우리의 위치를 파악하는 것이 Win Strategy 수립의 전제 조건입니다.

### 12.4 Fit Score의 의미
Fit Score는 0~100점 사이의 수치로 표현되지만, 이는 절대적인 합격/불합격 점수가 아닌 현재 딜의 리스크와 기회를 시각화한 지표입니다.

### 12.5 점수를 “결론”이 아닌 “대화 도구”로 쓰는 이유
점수 자체보다 "왜 이 점수가 나왔는가"에 대한 팀 내부의 정렬이 중요합니다. AI의 추천 점수와 사람의 판단 차이를 논의하는 과정에서 숨겨진 리스크와 승리 가설이 도출됩니다.

## 13. UI / UX 설계 원칙

### 13.1 고객 여정 가시성
사용자가 언제든지 자신이 현재 어떤 구매 맥락(Stage)에 있는지 인지할 수 있도록 최상단에 고정된 Stage Anchor를 배치합니다. 이는 단순한 진행률 표시가 아니라, 현재 우리가 던져야 할 핵심 질문이 무엇인지 상기시키는 역할을 합니다.

### 13.2 단계별 정보 밀도 조절
모든 기능을 한꺼번에 노출하여 인지 부하를 주지 않습니다. 현재 단계에서 가장 중요한 의사결정 도구(예: 고려 단계의 Qualification, 평가 단계의 Competitive Fit)를 우선적으로 배치하고, 불필요한 정보는 숨기거나 읽기 전용으로 전환하여 집중도를 높입니다.

### 13.3 툴팁 기반 가이드 설계
데이터 입력 시 사용자가 막막함을 느끼지 않도록 각 항목에 가이드 질문을 포함한 툴팁을 제공합니다. 이는 단순한 도움말을 넘어, 숙련된 프리세일즈 엔지니어의 사고 과정을 복제할 수 있도록 돕는 교육적 장치입니다.

### 13.4 평가 결과 시각화 원칙
정량적 점수는 항상 시각적 컨텍스트와 함께 제공됩니다. 예를 들어, Qualification 점수는 4분면(Quadrant) 차트에 매핑되어 비즈니스와 기술 사이의 균형을 한눈에 파악할 수 있게 하며, 솔루션 맵은 트리맵(Treemap)을 통해 시장 점유율과 리스크의 크기를 직관적으로 인지하도록 설계합니다.
